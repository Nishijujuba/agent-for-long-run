# AGENTS.md

## 仓库定位

本仓库面向 agent 协作生产一份约 10 页、约 8 分钟的演示文稿。演示主题是：税务管理员、程序员、数据分析师、算法工程师、平台运营工程师等知识工作者，如何使用 agent 加速工作，如何与 agent 协作把工作做扎实，如何设计和使用 agent 完成长程任务。

目标听众是普通工作者和需要改进工作流程的一线专业人员。他们关心“明天如何把 agent 用进工作”，对大模型底层算法细节兴趣有限。算法概念只在能解释实践边界时出现，例如用通俗语言说明注意力机制、上下文窗口和记忆机制为什么会影响长程任务。

这里的核心产物是一条可复用的内容生产流水线，单次聊天记录只作为过程材料：

1. 从 `resources/` 收集和理解原材料。
2. 从原材料和主题中提炼候选 ideas。
3. 通过人与 GPT/agent 的讨论，把候选 ideas 打磨成达成共识的 markdown 文档。
4. 基于确认完成的 ideas 生成最终 PPT、演讲稿和必要的备注。

## 目录约定

- `resources/`: 原材料目录。优先放技术视频总结 PDF、文章、会议记录、访谈记录等输入材料。Agent 读取这里时必须区分“来源原文”“摘要”“推断”。
- `ideas/`: 共识 ideas 目录。一个 idea 对应一个 markdown 文件。每个文件应表达一个清晰概念、一个工作场景或一个可进入 PPT 的论点。
- `README.md`: 面向人类的项目概览和快速入口。
- `AGENTS.md`: 面向 agent 的执行规则、术语、质量门槛和工作流。
- `待删除/`: 临时弃用材料的停放目录。需要移除文件时，先移动到这里，交给用户最终清理。

## 关键术语

- Agent: 能围绕目标调用工具、读取上下文、分解任务、持续产出中间结果的 AI 工作单元。直观类比是“会使用电脑和资料库的实习同事”，它需要目标、边界、检查点和验收标准。
- 长程任务: 不能靠一次问答稳定完成的任务。它通常包含多步依赖、资料积累、阶段验收和错误回滚，例如从多份 PDF 中提炼观点，再生成一份对特定听众有说服力的 PPT。
- 注意力机制: 大模型在生成内容时判断“当前该关注哪些上下文”的机制。可以把它类比成开会时翻阅桌面资料：资料越多，找到真正相关部分越难，遗漏、混淆和成本都会上升。PPT 中只需解释这个直觉，无需展开公式、模型结构或优化细节。
- 上下文窗口: 模型一次能看到的信息范围。它像人的工作台面积，资料可以放很多，但工作台有边界；超出边界的信息需要靠摘要、检索、文件、记忆或检查点重新带回任务。
- Idea: 已经被提炼出来的内容单元。它可以是一个概念、一个案例、一个协作模式、一个风险提醒或一页幻灯片的核心论点。一个好的 idea 应该能回答：它解决谁的什么问题，为什么可信，如何落地。
- 共识 idea: 经过用户与 GPT/agent 讨论后确认采用的 idea。共识并不等于绝对真理，它表示当前项目阶段愿意把它作为后续 PPT 生成的输入。
- 原材料: 放在 `resources/` 中的来源文件。原材料提供证据和启发，agent 引用其内容时应标明来源文件名，无法确认的内容要标为推断。
- PPT 骨架: 最终演示文稿的页级结构。10 页、8 分钟意味着每页平均约 48 秒，内容密度必须克制。
- 验收标准: 判断一个中间产物是否可以进入下一阶段的条件。形式可写成 checklist，例如“每个角色至少有一个真实工作场景”“每个核心论点有来源或清晰推理链”。

## 工作流

### 1. 盘点原材料

Agent 开始工作前先列出 `resources/` 中已有材料，记录文件名、主题、可能贡献的概念和明显缺口。PDF 内容如果还没有被提取成文本，先生成可追溯的阅读笔记，再进入 idea 提炼。

### 2. 生成候选 ideas

候选 ideas 可以由 agent 根据原材料和主题自动生成。每个候选 idea 需要包含：

- 标题
- 一句话主张
- 面向的角色或工作场景
- 来自哪些原材料
- 能进入 PPT 的理由
- 需要用户确认的问题

候选阶段允许粗糙，必须标清不确定性。

### 3. 讨论并确认 ideas

讨论时一次只解决一个关键问题。Agent 需要先给推荐答案，再说明取舍。若术语含糊，先定义术语；若角色场景过泛，拆成具体任务链路。

典型追问：

- 这个 idea 服务哪类听众？
- 它对应的真实工作任务是什么？
- Agent 在其中承担检索、写作、分析、执行、检查、编排中的哪一种角色？
- 人类需要在哪些节点做判断？
- 失败风险是什么？
- 这能否支撑一页 PPT？

### 4. 写入 `ideas/`

确认后的 idea 写入 `ideas/<短标题>.md`。建议模板：

```markdown
# <Idea 标题>

## 状态

candidate | confirmed | rejected

## 一句话主张

<用一句话说明这个 idea 的中心论点。>

## 适用角色

<税务管理员 / 程序员 / 数据分析师 / 算法工程师 / 平台运营工程师 / 跨角色>

## 工作场景

<描述一个具体任务，而非抽象口号。>

## Agent 协作方式

<agent 承担什么，人在什么节点介入。>

## 长程任务设计要点

<目标、资料、计划、检查点、记忆、工具、验收标准。>

## 来源依据

- `resources/<文件名>`: <相关内容摘要>

## 风险与边界

<隐私、合规、幻觉、权限、责任归属、质量控制等。>

## PPT 使用建议

<适合放在哪一页，配什么例子或图示。>

## 待确认问题

- <问题 1>
```

### 5. 生成 PPT 骨架

在 ideas 足够稳定后，先生成 10 页左右的 PPT 骨架，再写逐页内容。建议页型：

1. 标题页：Agent 作为知识工作的长期协作者。
2. 问题页：知识工作被材料、流程、反复沟通和质量检查拖慢。
3. 框架页：人负责目标、判断和责任；agent 负责检索、生成、执行和自检。
4. 税务管理员：政策检索、口径整理、风险提示和材料清单。
5. 程序员：需求澄清、代码修改、测试、代码审查和文档同步。
6. 数据分析师：数据口径、探索分析、图表解释和报告初稿。
7. 算法工程师：论文阅读、实验计划、评估设计和结果复盘。
8. 平台运营工程师：监控排障、工单归因、流程自动化和知识库沉淀。
9. 长程任务方法：目标、上下文、工具、检查点、记忆和验收；用“工作台面积有限”的类比解释上下文和注意力瓶颈。
10. 行动建议：从一个高频任务开始，建立可检查的 agent 协作流程。

### 6. 生成最终产物

最终产物可以包括：

- PPT 文件
- 演讲稿
- 每页 speaker notes
- 来源引用清单
- 生成过程记录

若创建输出目录，优先使用 `outputs/`。

## 质量门槛

- 每个核心论点必须有来源依据或明确推理链。
- 面向角色的内容必须落到具体任务，避免只写“提升效率”“自动化赋能”等空话。
- 长程任务必须包含检查点和验收标准。没有验收标准的长程任务容易变成堆材料。
- 技术解释必须服务普通人的工作理解。注意力机制、上下文窗口、记忆、检索、工具调用等概念可以讲，但讲法要落到“为什么长任务容易丢上下文、为什么需要分阶段验收、为什么要把关键结论写入文件”。
- 不应把篇幅花在 LLM 注意力优化、模型架构、训练细节、推理加速等算法细节上，除非这些细节直接影响普通工作者如何使用 agent。
- 涉及税务、合规、隐私、权限和生产系统操作时，必须写清 agent 的边界。Agent 可以辅助分析和整理，人类仍需承担最终业务判断。
- PPT 每页只承载一个主论点。8 分钟演讲不适合塞入过多工具名、模型名和抽象框架。
- Agent 生成内容后必须自检：事实是否有来源、角色是否具体、逻辑是否连贯、是否服务最终 PPT。

## Agent 执行规则

- 开始任务时先读取本文件和 README，再盘点 `resources/` 与 `ideas/`。
- 如果用户要求基于原材料生成 ideas，先列材料清单，再给候选 idea。
- 如果用户要求讨论某个 idea，一次只推进一个关键问题，并给出推荐答案。
- 如果用户确认某个 idea，及时写入或更新 `ideas/` 中对应 markdown。
- 不得编造 PDF 或视频中的内容。无法从材料确认时，标为“推断”或“待确认”。
- 不得直接删除文件。需要弃用文件时，移动到根目录 `待删除/`。
- 默认使用中文产出，除非用户要求英文或双语。
- 对非显而易见的选择，先回答“为什么选这个方案，替代方案是什么，代价是什么”。
