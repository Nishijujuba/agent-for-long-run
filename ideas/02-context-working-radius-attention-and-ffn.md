# 上下文有稳定工作半径：Attention 负责找关系，FFN 不负责验真

## 状态

confirmed

## 一句话主张

长上下文不是免费记忆；资料越多，关系越复杂，agent 越需要把任务切回模型的稳定工作区，并把关键事实落盘或交给工具验证。

## 适用角色

跨角色，尤其适合解释给普通知识工作者：为什么“把所有材料都塞进去”并不等于 agent 更可靠。

## 工作场景

税务管理员一次性塞入多年政策、口径讨论和客户材料；程序员一次性塞入整个仓库、错误日志和需求对话；数据分析师一次性塞入所有 SQL、图表和历史报告。用户以为上下文越多越安全，实际 agent 可能更容易混淆当前目标、旧结论和无关噪声。

## 讲给 PPT agent 的内容说明

这一页要把底层概念讲浅，但讲准。注意力机制可以类比成“在会议桌上找相关资料”。桌面越大，可以放的材料越多；但材料之间的关系也快速增长。可以用工程直觉公式说明：

`relations(n) ≈ n²`

`n` 是上下文里的 token 数量，关系压力随 `n` 增长得很快。这个公式不需要讲完整 Transformer，只用来解释：大上下文能放更多材料，却不会自动理清材料之间的依赖、冲突、时效和优先级。

FFN 可以理解成模型内部对局部模式做变换和组合的“熟练工”。它能把上下文变成流畅输出，但它不会天然知道某个历史结论是否已过期、某个测试是否真的跑过、某个政策是否适用于当前地区。因此长程任务必须用文件、检索、测试、日志、人工判断来补上验真机制。

## 实际例子

课程笔记里把 context window 比成会议室或工作台。参与者越多，维持“谁和谁相关、谁覆盖谁、谁改变了谁的意思”就越难。Matt Pocock 的经验判断是，coding 任务即使有很大的标称上下文，也要警惕上下文后段质量下降。压缩历史可以短暂续航，但摘要可能保留结论、丢掉证据；保留计划、丢掉反对意见；甚至把错误假设写得更像事实。

对于本仓库的 PPT 任务，安全做法不是把所有 PDF、聊天记录和仓库文件塞进一个 prompt，而是让 agent 逐份读取、提炼 idea、写入 `ideas/`，再由后续 agent 读取稳定文件继续。

## 应掌握的概念和方法论

- Smart Zone：信息量适中、目标单一、关系清楚时，模型判断较稳。
- Dumb Zone：上下文堆积、目标混合、历史修补太多时，模型仍会流畅输出，但取舍开始漂移。
- Clear：清空会话，让下一轮从干净状态开始。
- Compact：压缩历史，只适合临时续航；不能把摘要当作完整证据。
- Retrieval：只把当前任务需要的证据取进上下文，其余留在文件和索引里。
- State file：当前状态放文件里，聊天历史只做参考。

## 常见失败方式

第一种失败是“全文投喂崇拜”：用户把资料越塞越多，以为模型一定会综合得更好。结果是 agent 记住了很多词，却丢了当前目标。

第二种失败是“摘要信任过度”：经过多轮 compact 后，摘要看起来更清晰，却可能已经丢掉关键反例、过期状态和来源证据。

第三种失败是“让模型自己验真”：模型能生成解释，但测试是否真的通过、数据库是否真的更新、政策是否真的适用，都必须回到工具和人。

## PPT 使用建议

适合做“为什么长程任务会跑偏”的基础页。画面可以用一张工作台：左侧是少量清晰资料，右侧是堆满 PDF、日志、旧计划和截图的混乱桌面。核心不是讲模型架构，而是让听众明白：长程任务要管理信息进入上下文的方式。
