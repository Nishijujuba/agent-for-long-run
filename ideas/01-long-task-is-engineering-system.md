# 长程 Agent 不是更长 Prompt，而是工程系统

## 状态

confirmed

## 一句话主张

长程任务的关键不是让模型“一次想得更久”，而是把聪明模型放进一个可恢复、可验证、可限权、可观测的工程系统里。

## 适用角色

跨角色：税务管理员、程序员、数据分析师、算法工程师、平台运营工程师。

## 工作场景

一个人想让 agent 完成“读多份材料、提炼观点、写 ideas、再生成 8 分钟 PPT”这类任务。它不是单次问答，而是包含资料读取、观点筛选、任务拆分、文件落盘、质量检查、后续交接的长链路任务。

## 讲给 PPT agent 的内容说明

这一页应建立总判断：长程 Agent 的成败更像工程系统稳定性问题，而不是单个模型能力问题。可以用公式表达：

`Agent = LLM × Harness × Memory × Tools`

LLM 负责理解、生成、权衡；Harness 负责循环、权限、预算、日志、停止条件；Memory 负责状态、长期记忆、检索和交接；Tools 负责测试、搜索、文件读写、数据库查询、浏览器检查等外部证据。

可以补一个很直观的失败概率解释。若单步成功率为 `p`，任务需要 `T` 步，则全链路成功率可粗略写成：

`P(success) ≈ p^T`

即使每一步 95% 可靠，50 步后也只剩约 7.7%。所以长程任务不能靠“模型很聪明”硬扛；每一步都要接回外部证据，比如测试结果、日志、diff、截图、数据库返回、人工审查。

## 实际例子

本仓库本身就是一个长程任务：`resources/` 放多份 PDF，`ideas/` 放讨论后的共识 idea，后续再生成 PPT。若 agent 只在聊天窗口里“总结一下所有资料”，很容易丢来源、混淆结论、产出空泛观点。更稳的做法是把任务分成：盘点资源、提炼候选 idea、逐个确认、写入 markdown、再让另一个 agent 基于这些文件生成 PPT。

## 应掌握的概念和方法论

- 任务契约：目标、非目标、资料来源、验收标准、权限边界。
- 状态文件：把当前进度、已确认决策、证据、风险、下一步落到文件里。
- 外部验证：测试、lint、typecheck、构建、浏览器 smoke check、人工 QA。
- 小步循环：计划、行动、观察、修正；每轮都要留下可审查证据。
- 人类责任：agent 可以执行和检查，人类仍负责业务判断、合规风险、最终质量。

## 常见失败方式

最典型的失败是把“生成一份 PPT”这类大目标直接扔给 agent，让它在一个长会话里边读资料边规划边写作边自检。表面上输出很流畅，实际缺少来源约束、缺少状态、缺少验收，后续 agent 也无法接手。

另一个失败是把权限、预算、停止条件都交给模型自己判断。模型会倾向于继续尝试、继续修补、继续解释，最后形成高成本、低证据、难回滚的长链路。

## PPT 使用建议

适合做总框架页。画面可用“生产线”或“安全驾驶系统”类比：模型像发动机，harness 像方向盘、刹车、仪表盘和道路护栏。不要把这一页写成工具清单；重点讲清楚为什么长程任务必须系统化。
