# 04 任务图：PRD → Kanban/DAG，把目标变成可抓取工作项

## 状态

confirmed

## 一句话主张

长程任务需要把“完成后的形状”和“抵达路径”分开：PRD 描述目的地，Kanban/DAG 描述旅程，agent 才能拿到清楚、可验收、可排序、可回滚的工作项。

## 适用角色

跨角色：程序员、数据分析师、算法工程师、平台运营工程师；税务和合规场景也可以把政策口径拆成待确认、可执行、可复核的任务项。

## 工作场景

一段对齐讨论产生很多共识：用户故事、指标口径、代码边界、测试决策、上线限制。若直接让 agent 实现，目标和路线会混在一起。更稳的做法是先生成 destination document，再拆成本地 issue、Kanban 或 DAG。

## Agent 协作方式

人类检查 PRD 是否表达真实目的地，检查 ticket 是否是垂直切片。Agent 负责把 PRD 转成可抓取任务，识别依赖关系，按已解锁任务循环执行、测试、提交和总结。

## 长程任务设计要点

- Destination document：描述完成后的形状，例如用户能看到什么、系统要支持什么、哪些内容范围外。
- Journey document：描述抵达路径，例如任务顺序、阻塞关系、并行关系、回滚边界。
- Kanban 字段建议包含：`type`、`blocked by`、验收方式、风险等级、可回滚边界。
- DAG 的价值在于暴露阻塞和并行关系，让 agent 只抓取已解锁任务。
- 垂直切片优先：每张 ticket 尽量贯穿必要层次，早一点产生真实反馈。

## 风险与边界

任务拆分过粗会让 agent 背负整个世界；任务拆分过细且只按层拆，会延迟集成反馈。人类需要检查切片质量，尤其要防止“先做所有数据库、再做所有 API、最后接前端”这类横向铺层。

## PPT 使用建议

**页面标题**：PRD → Kanban/DAG：把目标变成可抓取工作项

**页面主文案**：

- PRD：完成后的形状，包含用户故事、DoD、范围外。
- Ticket A：AFK、无阻塞、端到端薄片。
- Ticket B：Human-reviewed，口径需确认。
- Ticket C：AFK，blocked by A。
- DAG：并行与阻塞关系可审查。
- Agent Loop：执行、测试、提交、总结。

**讲法**：用导航类比。PRD 是终点地址，Kanban/DAG 是路线图。没有终点，agent 会乱开；没有路线图，agent 会把所有路一起走。
