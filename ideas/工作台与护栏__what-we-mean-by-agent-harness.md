# Agent Harness：给长程任务配工作台与护栏

## 状态

candidate

## 一句话主张

Agent 能稳定完成长程任务，关键在于提前设计 Harness：工具环境、自由度、约束、状态记录、执行循环和交叉检查，让 agent 像进入一间有工具、有规程、有交接本、有复核人的工作间。

## 适用角色

跨角色，尤其适合程序员、数据分析师、算法工程师和平台运营工程师。税务管理员也可用于政策检索、口径整理和材料清单生成等低权限辅助任务。

## 工作场景

以下平台运营排障案例为面向本项目听众的分析推断，PDF 未直接给出该业务场景。

以平台运营工程师处理一次线上告警为例：agent 先读取告警、日志、变更记录和历史工单，提出排障假设，生成排查步骤和复盘草稿；另一个 reviewer agent 交叉检查证据、风险和遗漏；人类只在高风险节点做判断，例如是否执行生产变更、是否对外发布结论。

## Agent 协作方式

- 人类设定目标、权限边界、验收标准和停机点，例如“只读日志和工单，不执行生产操作”。
- Harness 提供工具环境，例如日志查询、工单系统、代码库、浏览器、文档库，以及每个工具的可用范围。
- 执行 agent 负责检索、分析、生成草稿和记录过程。
- 复核 agent 负责 Cross-check：检查证据是否支撑结论，指出遗漏，必要时 veto 或 stop。
- 人类在检查点确认方向、承担业务责任，并把关键判断写入任务记录，避免上下文漂移。

## 长程任务设计要点

- 目标：把任务写成可验收的交付物，例如“输出一份告警归因报告，包含证据链、影响范围、修复建议和未确认风险”。
- 资料：列出来源清单，区分日志事实、历史案例、agent 推断和人类判断。
- 工具：为 agent 配置必要工具，优先只读权限；写入、外发、生产变更等动作需要人类确认。
- 自由度：允许 agent 在资料内探索多条假设，限制它越权访问、擅自执行高风险动作。
- 约束：给出隐私、合规、生产安全、引用来源和输出格式要求。
- 状态管理/任务记忆：保留任务台账，用于跨轮次保存当前假设、已查证据、已排除原因、人工判断、待确认问题和下一步。
- 检查点：计划确认、敏感数据访问前确认、结论发布前复核、生产变更前人工批准。
- 编排循环：采用“计划 -> 执行 -> 观察 -> 记录 -> 复核 -> 人类确认 -> 下一轮”的节奏。
- 上下文预算：PDF 图中给出一个直觉模型，当上下文使用比例 \(r=\frac{C_{used}}{C_{max}}\) 超过约 \(0.5\) 时，有效能力可能下降，可用 \(I_{effective}\approx I_0 e^{-\lambda(r-0.5)}\) 表示这种衰减趋势。PPT 中应把它讲成“工作台堆满资料后，人和模型都会更容易漏看重点”的警戒线。
- 验收标准：结论有来源，复核 agent 无高优先级异议，人类确认风险边界，所有待确认项明确列出。

## 风险与边界

- Cross-check 能降低遗漏风险，无法保证结论正确。复核 agent 也是模型输出，仍需来源证据和人类判断。
- Multi-agent 会增加成本、延迟和协调复杂度，适合高风险或长程任务；简单任务可使用单 agent 加人工抽查。
- Harness 设计过宽会放大越权和误操作风险，设计过窄会让 agent 只能写空话。权限边界需要随任务风险分级。
- 税务、合规、隐私和生产系统操作必须保留人类最终责任。Agent 可整理材料、提示风险、生成草稿，业务确认和对外动作由人承担。
- Self-Evolution 需要可审计反馈。若把用户偏好、业务秘密和错误样例无边界地沉淀进记忆，会带来隐私泄露和错误固化。

## PPT 使用建议

适合放在第 9 页“长程任务方法”，也可作为第 3 页“人和 agent 分工框架”的方法页。

建议画一张简单流程图：左侧是人类设定目标与边界，中间是 Harness 的六个模块：工具、自由度、约束、状态、循环、复核，右侧是可验收交付物。类比可以用“给临时同事安排工位”：钥匙给哪些、资料放哪里、交接本怎么写、谁来复核、什么情况必须停下问负责人。

页面主句建议：长程任务的稳定性，来自 agent 之外的工作设计。
