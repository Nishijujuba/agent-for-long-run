# 03 Human-in-the-loop：任务对齐先用文档拷问需求

## 状态

confirmed

## 一句话主张

长程任务的规划阶段必须让人类留在环里：agent 可以带着文档追问、给推荐答案和证据，人类负责做价值取舍、领域判断和风险确认。

## 适用角色

跨角色：尤其适合政策口径整理、产品需求、代码修改、指标报告、实验方案、运维排障等需要先对齐判断口径的任务。

## 工作场景

用户说“帮忙做一个功能”或“整理这个口径”，agent 先读取相关资料，再一次只问一个关键问题，例如适用地区、生效日期、指标定义、模块边界、上线权限。每个问题都带推荐答案、依据和替代选项，便于人类快速拍板。

## Agent 协作方式

Agent 承担“带资料追问”的角色：把模糊 brief 拆成决策树，把每个问题连接回文档、代码或业务证据。人类承担“裁判和责任人”的角色：确认取舍、否定错误假设、明确不可越界内容。对齐结束后，agent 才把共识压缩成 PRD、issue 或状态文件。

## 长程任务设计要点

- 一次只推进一个关键决策，避免把多个未确认问题混进同一个回答。
- 每个问题给出推荐答案，让人类有可反驳的默认项。
- 记录已确认决策、反对理由、证据来源和待确认边界。
- 区分 Human-in-the-loop 任务与 AFK 任务：规划、价值判断、合规风险需要人类；目标和验证方式清楚的实现任务可以进入离键执行。

## 风险与边界

最大风险是 agent 把未确认的业务判断写成事实。税务、合规、权限、生产操作、用户影响范围等内容必须由人类确认。Agent 的推荐答案可以提高讨论效率，不能替代责任归属。

## PPT 使用建议

**页面标题**：任务对齐：先用文档拷问需求

**页面主文案**：

- Grill with docs：agent 带着资料追问。
- 一次一问：沿决策树推进，避免混杂判断。
- 推荐答案：给人类一个可反驳默认项。
- 证据来源：回答要能回到文档、代码、日志或数据。
- 待确认边界：把风险留在人工判断处。

**讲法**：用“agent 像拿着资料夹的项目助理”作类比。它负责把问题问尖，把证据摆出；人类负责最后取舍。对齐完成后，才有资格把对话变成 PRD 或任务清单。
