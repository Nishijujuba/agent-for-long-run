# 先做任务契约：长程任务不是直接开跑，而是先把终点说清楚

## 状态

confirmed

## 一句话主张

让 agent 做长程任务前，最重要的不是写更长的 prompt，而是先把目标、范围、资料、权限、检查点和验收标准变成一份可执行的任务契约。

## 适用角色

跨角色：税务管理员、程序员、数据分析师、算法工程师、平台运营工程师。

## 工作场景

用户说“帮我读这些材料并做 PPT”“帮我实现一个功能”“帮我分析这批数据”。这些请求听起来明确，但实际隐藏了很多关键选择：最终产物长什么样、哪些材料可信、哪些内容不能做、什么时候要问人、什么证据证明已经完成。

## 讲给 PPT agent 的内容说明

这一页应该讲清楚：长程任务的第一步不是让 agent 立刻执行，而是让它带着资料反问，把模糊 brief 转成任务契约。契约不是形式主义，它是后续所有自动化的边界。

可以把任务契约解释成“给 agent 的施工许可证”：它说明目的地、施工范围、可用工具、禁止事项、阶段验收和停工条件。没有这份契约，agent 会把“继续努力”误当成“接近完成”，最终产出可能很流畅但不可验收。

人类在这一页里的角色要说清楚：agent 可以读取材料、提出问题、给推荐答案、整理取舍；人类负责业务判断、风险偏好、合规边界、审美和最终责任。

## 应掌握的概念和方法论

- Task contract：目标、非目标、输入资料、输出格式、验收标准、时间/成本预算、权限边界。
- Recommendation-backed question：agent 不只是问“你想怎样”，而是带着证据给出推荐选项和代价。
- Human-in-the-loop：需要价值判断、业务后果、合规风险和最终质量时，人必须在环。
- Negative decisions：被明确拒绝的方案也要记录，防止后续 agent 重新捡起来。
- AFK-ready task：只有目标、边界和验收都清楚后，任务才适合离键交给 agent 执行。

## 常见失败方式

最常见的失败是把模糊需求直接交给 agent。它会自动补全缺失假设，例如自行决定指标口径、功能范围、政策适用边界或 PPT 受众重点。输出看似完整，实际是模型替人做了未经授权的业务判断。

另一个失败是追问很多但没有收敛。好的澄清不是问题清单越长越好，而是把关键分歧变成已确认决策、开放问题和验收标准。

## PPT 使用建议

适合做“方法总入口”页。画面可以是一条转换链：模糊请求 -> 任务契约 -> 可执行 backlog -> agent 执行。重点不要写成“写好 prompt”，而要写成“先设计任务边界”。
