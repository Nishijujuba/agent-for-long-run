# 可控循环：让 Agent Loop 在预算和检查点内迭代

## 状态

candidate

## 一句话主张

Agent loop 的价值在于让 agent 看见中间结果后继续推进；在知识工作中，它需要被放进“目标、规格、预算、检查点、验收”的框架里，避免错误假设和 token 成本在多轮循环中被放大。

## 适用角色

跨角色。主例适合程序员，迁移到数据分析师、平台运营工程师、算法工程师也成立。

## 工作场景

程序员要完成一个内部工具的小功能改造，例如给报表系统增加导出功能。人先把目标写成 PRD 或 Markdown spec，定义验收标准和最多循环轮数；agent 每一轮读规格、改代码、查看测试或运行结果，再决定下一步。人类在需求解释、权限授权、关键结果验收处介入。

## Agent 协作方式

- 人类负责：写清 `/goal`、PRD、Markdown spec，规定检查点、预算和不可越过的边界。
- Agent 负责：读取目标和规格，执行一轮任务，观察 Result，把结果和规格对照，继续下一轮。
- 人类检查点：每轮结束查看差异、测试证据、假设列表和下一步计划。
- 直观类比：这像让一位同事按返工单连续修改。返工单写得清楚，他能自己推进；返工单含糊，他会猜，猜错后还会继续沿着错路返工。

## 长程任务设计要点

- 目标：用一句话定义最终交付物，例如“实现导出功能，并通过现有测试”。
- 资料：把 PRD、spec.md、接口约束、样例数据、测试命令放入可引用文件。
- 计划：每轮遵循“执行 -> 看结果 -> 对照规格 -> 记录假设 -> 决定下一步”。
- 检查点：每轮输出结果、证据、风险、下一步，避免只给出完成宣称。
- 记忆：把重要判断、用户确认、失败原因写入文档或任务记录，防止循环中丢上下文。
- 工具：程序员场景可接入 Cursor、GitHub、Greptile、测试命令；非编程场景可替换为数据查询、文档检索、表格分析、工单系统。
- 预算：用公式表达为 \(T \le B\)。其中 \(T\) 是实际消耗的 token、时间、API 调用或循环轮数，\(B\) 是预先设定的上限。
- 验收标准：功能通过测试、结论有来源、假设被列出、高风险动作经过人类批准。

## 风险与边界

- 假设风险：agent 在信息缺口处会补假设；假设若未被记录和检查，后续循环会扩大偏差。
- 成本风险：循环会消耗 token、时间和 API 预算，循环轮数需要上限。
- 责任边界：税务、合规、隐私、生产系统操作等场景中，agent 可以整理、检查、模拟，人类仍承担最终业务判断和授权。
- 质量边界：没有 PRD、spec、测试或验收清单时，循环容易变成“看起来很忙”的自动返工。
- 上下文边界：长程任务要把目标、决策、失败原因外化到文件；只放在聊天上下文里，后续循环容易遗忘或混淆。
