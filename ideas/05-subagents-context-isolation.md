# 05 分工与并行：Sub-agent 用上下文隔离扩大探索半径

## 状态

confirmed

## 一句话主张

Sub-agent 的核心价值是上下文隔离：父代理保留主线，子代理去资料库、代码库、评审室或风险清单里探索，最后只把摘要、证据和建议带回主任务。

## 适用角色

跨角色：复杂代码库改动、多资料政策检索、数据口径审查、实验方案对抗评审、平台事故排查。

## 工作场景

平台运营工程师要排查一次告警。主 agent 负责保持事件主线和下一步决策；一个子 agent 读日志，一个子 agent 查近期变更，一个子 agent 找类似事故 runbook，一个子 agent 做反例检查。各自回传结论后，主 agent 再合并判断。

## Agent 协作方式

父代理像项目经理，负责目标、优先级和最终判断。子代理像被派到档案室、测试场、评审室的研究员，负责大范围读取、候选方案整理、漏洞寻找、风格变体或已解锁任务并行执行。

## 长程任务设计要点

- 探索类任务适合交给子代理：读仓库、读资料、找历史、列方案。
- 评审类任务适合放进独立上下文：让 reviewer 离开主会话噪音，专注检查漏洞。
- 并行类任务必须基于 DAG，只让子代理抢已解锁、低耦合、验收清楚的任务。
- 子代理回传摘要需要证据锚点；高风险结论必须回到源文件、代码、日志或测试里抽查。

## 风险与边界

多代理会提高探索覆盖面，也会提高协调成本。摘要可能遗漏边界，多个子代理也可能基于不同假设行动。父代理需要维护统一状态文件、任务图和证据要求；涉及写操作、权限、生产系统时，子代理默认从只读和沙箱开始。

## PPT 使用建议

**页面标题**：Sub-agent：隔离上下文，扩大探索半径

**页面主文案**：

- Orchestrator：保留主线。
- 探索代理：读仓库、读资料，回传摘要。
- 对抗评审：找漏洞、反例、合规风险。
- 并行代理：抢已解锁任务，按 DAG 加速。
- 风格覆盖：保守版、激进版、业务版、技术版。
- 高风险摘要要回到证据源抽查。

**讲法**：用“父代理像项目经理，子代理像研究员”作类比。真正的收益来自隔离噪音和扩大探索半径，随后由主线统一决策。
