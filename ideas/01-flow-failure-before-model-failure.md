# 01 问题定义：长程任务的失败，通常先发生在流程

## 状态

confirmed

## 一句话主张

人们使用 agent 做长程任务时，失控点常来自流程：目标没对齐、上下文越堆越重、验证来得太晚，会让 agent 流畅地产出偏离目标的结果。

## 适用角色

跨角色：税务管理员、程序员、数据分析师、算法工程师、平台运营工程师，以及需要把多资料、多步骤、多轮检查串起来的知识工作者。

## 工作场景

一个人让 agent 整理政策口径、修改功能、生成数据报告或复盘实验。任务一开始只有模糊口号时，agent 会一边猜目标、一边扩展材料、一边生成结果。短任务里这种猜测看起来像效率，长任务里它会变成偏差复利。

## Agent 协作方式

人类先定义目标、范围外、责任边界和验收标准；agent 负责读取材料、提出风险、拆解任务、生成中间稿并自检。流程像护栏，会把 agent 的能力放到可控轨道上。

## 长程任务设计要点

- 先写任务契约：目标、范围外、资料来源、责任边界、验收标准。
- 把大任务拆成可验收阶段；背景材料通过文件、索引和状态记录分层进入上下文。
- 每阶段都要留下可追踪的决策痕迹：为什么这样做、依据是什么、还有哪些未确认问题。
- 越接近业务判断和真实用户影响，越需要人工验收。

## 风险与边界

最大的风险是把 agent 当成魔法助手，把流程问题误判成模型问题。模型越流畅，越容易掩盖早期目标偏差。涉及高影响任务时，agent 可以辅助整理和检查，人类仍然承担最终判断。

## PPT 使用建议

**页面标题**：长程任务的失败，通常先发生在流程

**页面主文案**：

- 目标没对齐：大家以为懂了，脑中画面其实不同。
- 上下文越堆越重：历史、日志、讨论、修补说明互相干扰。
- 验证来得太晚：做完很多层之后，才看到真实结果。
- 结论：agent 的流畅输出会掩盖偏差；偏差进入文件、代码、报告后，修复成本迅速上升。

**讲法**：这页要直接打破幻想。长程 agent 任务的抓手是把目标、资料、步骤、检查和责任固定在流程里。模型负责生成和推理，流程负责防止偏差一路滚大。
