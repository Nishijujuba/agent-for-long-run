# 业务验收尺：用小基准集验收 Agent

## 状态

candidate

## 一句话主张

让 agent 进入真实工作前，先用 5-10 个代表性业务样本建立“小基准集”，用可复测分数判断 prompt、模型、工具和流程是否真的可靠。

## 适用角色

跨角色。主例适合数据分析师和平台运营工程师；税务管理员、程序员、算法工程师也可以把历史任务改造成同类验收样本。

## 工作场景

数据分析师每周让 agent 生成运营异常分析报告。团队先挑 8 个历史案例：有的是真异常，有的是口径变化，有的是节假日波动，有的是埋点错误。每个案例配好输入材料、标准答案要点和评分规则。之后每次改 prompt、换模型、加工具或启用记忆，都先跑这 8 个案例，再决定是否进入正式报告流程。

## Agent 协作方式

- 人类负责挑样本、写评分标准、确认业务风险。样本要像体检项目，覆盖常见病、疑难病和容易误判的边界情况。
- Agent 负责按固定流程处理每个样本，输出报告、证据引用、风险提示和未确认问题。
- 评审 agent 或脚本按 checklist 初评，例如事实正确、引用完整、口径一致、风险提示充分、格式可交付。
- 人类抽查低分样本和高风险样本，决定 prompt、模型或工具链是否放行。
- 每次真实工作出现新错误，就把它沉淀成下一个 benchmark case，形成“失败样本库”。

## 长程任务设计要点

- 目标：把“感觉 agent 好用”转成“在本业务样本上可复测地达标”。
- 资料：历史工单、报告、政策问答、代码任务、实验记录、异常日志都可以成为样本；敏感内容要脱敏。
- 计划：先建样本，再建评分表，然后固定运行流程，最后比较 baseline 与新方案。
- 检查点：样本选择检查、评分标准检查、运行结果检查、人工抽查检查、上线放行检查。
- 记忆：把样本、标准答案、评分规则、失败原因和修复记录写入文件，避免依赖一次聊天历史。
- 工具：可使用表格、脚本、测试框架、日志查询、文档检索、报告模板和评审 agent。
- 验收标准：新方案平均分高于 baseline，关键样本不能退步，高风险项必须有人类签字。
- 可用简化公式表达验收尺：

\[
B_{\text{work}}=\frac{1}{n}\sum_{i=1}^{n}s_i
\]

其中 \(n\) 是样本数量，建议从 5-10 个开始；\(s_i\) 是第 \(i\) 个样本的得分。这个公式像一把尺子：尺子很短也能先量出方向，后续再逐步加长。

分析推断：

- PDF 的讲法偏 agent 工程实践；面向普通知识工作者时，最适合转译成“业务验收尺”。
- 5-10 个样本不是最终充分覆盖，它更像启动门槛，用来避免只凭 demo 和主观感受上线。
- 实际使用时，5-10 只适合作为启动规模；高风险业务应增加样本数量、重复运行次数和人工复核比例。
- 数据分析、税务政策整理、平台排障、代码修改、算法实验复盘都可以采用同一结构：历史样本、标准答案、评分表、baseline、新方案对比。

## 风险与边界

- 小基准集会遗漏长尾问题。它适合做第一道门槛，不能替代真实业务复核。
- 样本选得太容易会制造虚假安全感。必须加入边界样本、失败样本和反例。
- 评分规则如果只看格式，agent 会学会写漂亮废话。评分要包含事实、证据、口径、风险和可执行性。
- 税务、合规、生产系统和客户影响场景必须保留人类最终判断。Agent 可以跑样本、生成对比、提示风险，放行责任仍在人。
- Benchmark 本身也会过期。政策、业务口径、系统结构、用户行为变化后，样本和标准答案都要更新。

## PPT 使用建议

适合放在第 3 页“人和 agent 分工框架”或第 9 页“长程任务方法”。页面主句建议：

“先给 agent 一把业务尺，再让它进真实流程。”

视觉上可以画一条简单流水线：

`历史样本 -> 标准答案 -> agent 运行 -> 自动评分 -> 人工抽查 -> 放行 / 回炉`

讲解例子优先用数据分析师的异常报告，因为普通工作者容易理解“拿历史题考试”。可补一句迁移：税务管理员用历史政策问答，平台运营工程师用历史故障单，程序员用旧 bug 和测试用例。
