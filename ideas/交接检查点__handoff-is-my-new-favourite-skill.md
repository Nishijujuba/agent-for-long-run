# 交接检查点：把长程任务切成可接手的工作包

## 状态

candidate

## 一句话主张

长程 agent 任务需要“交接检查点”：当主线任务出现旁支、原型验证、审查或信息核实时，把相关上下文压缩成短命、聚焦、可执行的 handoff 文档，交给新会话处理，主会话保留目标、判断和责任。

## 适用角色

跨角色：税务管理员、程序员、数据分析师、算法工程师、平台运营工程师。

## 工作场景

这些跨角色例子属于本项目迁移推断，PDF 原例主要集中在 coding、grilling、prototype、GitHub issue、review 等 agent 协作场景。

一个知识工作者正在和 agent 做长任务，例如整理税务政策口径、排查平台告警、分析业务数据、规划代码改造或阅读论文做实验计划。任务进行到一半时，agent 发现一个旁支问题：

- 税务管理员：某条政策口径需要单独核对原文、地区差异和材料清单。
- 程序员：当前需求之外出现一个重构机会或单独 bug。
- 数据分析师：主报告之外发现一个指标口径需要追溯数据来源。
- 算法工程师：论文方案之外需要做一个小实验确认可行性。
- 平台运营工程师：当前工单之外发现一个可能的系统性告警归因。

此时继续把旁支塞进主会话，会占用同一个上下文窗口和注意力预算。更稳的做法是生成 handoff 文档，把旁支变成一个可交给新 agent 的工作包。

## Agent 协作方式

- 人类负责确定主线目标、旁支是否值得拆出、最终业务判断和风险责任。
- 主会话 agent 负责识别旁支、提取任务切片、写出 handoff 文档，并说明“为什么交出去、交给谁、完成什么”。
- 新会话 agent 负责根据 handoff 文档独立执行，例如核对资料、写 issue、做 prototype、跑审查或整理发现。
- 回程 handoff 负责把新会话的关键学习、证据、风险和下一步压缩回主会话。

可以用一个简单公式表达：

\[
H=\mathrm{select\_and\_compress}(C, I)
\]

其中：

- \(C\)：当前会话可见的完整上下文，包括对话、工具结果、文件状态和已有判断。
- \(I\)：下一会话的焦点，例如“核对这条政策口径”“为 API 拆分创建 issue”“验证这个 UI 原型”。
- \(H\)：最终 handoff 文档，只保留服务于 \(I\) 的必要信息、外部指针和下一步动作。

直观类比：handoff 像交班单。交班单不追求复述整天发生的所有事情，它要让接班人知道当前状态、关键证据、下一步和风险点。

## 长程任务设计要点

- 目标：先写清主线目标，再判断旁支是否会干扰主线。
- 资料：handoff 只放必要摘录，已有 PRD、issue、政策原文、数据字典、日志文件和代码文件用路径或链接指向。
- 计划：主会话继续推进原任务，新会话处理旁支任务，二者拥有各自的完成条件。
- 检查点：每次 handoff 都要写明接收方需要交回什么，例如“结论、证据、风险、建议动作、未解决问题”。
- 记忆：handoff 是短命交接材料，适合放临时目录；长期知识应沉淀到正式文档、知识库、issue 或报告。
- 工具：接收方可以按任务选择检索、代码审查、原型验证、数据核对、网页查证或安全审查工具。
- 验收标准：handoff 文档至少回答五个问题：下一会话焦点是什么，最小上下文是什么，哪些外部材料需要看，建议使用什么技能或执行方式，哪些敏感信息不能进入文档。

## 风险与边界

- handoff 文档可能泄露敏感信息。API key、password、token、cookie、客户数据、个人邮箱、手机号、内部 URL 和未公开业务信息应脱敏或改用受控来源指针。
- handoff 文档容易变成第二套文档。若复制 PRD、issue、政策原文或日志全文，后续很快会与正式材料冲突。应优先使用路径、链接和最小摘录。
- handoff 不能替代人类判断。税务、合规、隐私、生产系统操作和对外发布仍需人类确认责任边界。
- 过度 handoff 会制造协调成本。只有当旁支会污染主线、需要独立验证、需要另一种技能范式或需要长时间运行时，拆出才值得。
- 回程 handoff 不能只写“已完成”。它应交回非显而易见的学习结果、证据、失败路径、剩余风险和建议动作。

## PPT 使用建议

适合放在“长程任务方法”页，也可以作为全 deck 的方法论页。该页只承载一个主论点：长任务要靠可检查的交接协议保持清晰。

建议页面结构：

- 标题：把旁支交出去，让主线保持清楚
- 左侧图示：主会话 \(C\) 提取任务切片 \(C_{task}\)，结合焦点 \(I\)，生成 handoff 文档 \(H\)，启动新会话 \(S'\)
- 右侧 checklist：
  - 焦点：下一会话处理什么
  - 指针：要看哪些材料
  - 产物：交回什么结果
  - 边界：哪些信息不能写入
  - 验收：谁做最终判断
- 口播例子：税务管理员整理政策口径时，主会话继续做报告结构，旁支会话单独核对一条政策的地区差异和材料清单，最后只把结论、依据和风险提示交回主会话。

PPT 页建议优先使用中文 checklist；公式 \(H=\mathrm{select\_and\_compress}(C,I)\) 可放进备注或删减，避免普通听众被符号分散注意力。
