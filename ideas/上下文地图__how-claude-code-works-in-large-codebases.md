# 上下文地图：先给 agent 一张工作地图

## 状态

candidate

## 一句话主张

长程任务开始前，应先给 agent 准备“全局规则 + 工作地图 + 当前任务”三层上下文，让它先知道边界、入口和验收方式，再进入具体执行。

## 适用角色

跨角色，尤其适合程序员、数据分析师、平台运营工程师，也可迁移到税务管理员和算法工程师的资料型长任务。

## 工作场景

一个程序员要让 agent 修改大型代码库中的退款逻辑；一个数据分析师要让 agent 基于多张表生成经营分析；一个税务管理员要让 agent 整理一批政策文件中的适用口径。

这些任务共同的问题是：材料多、模块多、边界多。直接把任务丢给 agent，agent 容易在无关材料里搜索，遗漏关键约束，或把局部规则当成全局规则。

程序员案例可以拆成 3 步任务链：定位退款入口 -> 修改金额计算逻辑 -> 跑测试并记录影响范围。数据和税务场景也应先找入口，再执行任务，最后回到验收标准。

## Agent 协作方式

人类先准备两类文件：

- 规则文件：说明项目目标、权限边界、输出格式、检查点和禁止事项，类似 PDF 中的 `CLAUDE.md`。
- 地图文件：说明材料或系统的模块分布、每个模块负责什么、常见入口在哪里，类似 PDF 中的 `CODE_MAP.md`。

Agent 执行时先读规则和地图，再根据当前任务进入相关模块。人在关键节点检查：任务理解是否正确、选中的模块是否正确、产出是否满足验收标准。

直观类比：新同事第一天接手复杂项目时，先看公司手册和楼层导览图，再去具体办公室办事。公式可以写成：

\[
I(p)=C_{root}(R)\cup C_{module}(p)\cup C_{task}
\]

其中 \(C_{root}\) 是全局规则，\(C_{module}\) 是当前模块规则，\(C_{task}\) 是本次任务要求。

## 长程任务设计要点

- 目标：先定义本次任务要解决的问题，例如“修复退款金额计算错误”或“整理某类政策的适用条件”。
- 资料：把原始材料分层，区分全局规则、模块地图、当前任务证据。
- 计划：让 agent 先说明会查哪些模块、为什么查这些模块、暂时不查哪些模块。
- 检查点：至少设置三个检查点：任务理解确认、材料范围确认、最终结果验收。
- 记忆：把稳定规则写入规则文件，把结构性知识写入地图文件，把临时判断写入任务记录。
- 工具：程序员场景可结合 LSP、测试命令、代码搜索；数据和运营场景可结合表格、日志、监控、知识库检索。
- 验收标准：产出必须说明依据来源、影响范围、未覆盖风险和下一步人工确认事项。

地图的价值可以用 PDF 中的公式表达：

\[
S_{map}(T) \ll S_{full}(T)
\]

也就是说，一张结构地图的信息量远小于完整材料集，却能显著减少 agent 的无效搜索。

## 来源依据

- `resources/how-claude-code-works-in-large-codebases.pdf`

PDF 明确内容：

- PDF 主题围绕 Claude Code 在大型项目中的实践，目录包含 Harness、`CLAUDE.md`、Hooks、Skills、Plugins、LSP、MCP、Subagents、Code Map 等内容。
- PDF 给出分层上下文公式：`I(p)=C_root(R) ∪ C_module(p) ∪ C_task`。
- PDF 给出 Code Map 压缩项目结构的公式：`S_map(T) << S_full(T)`。
- PDF 给出组合公式：`C_root(R) + M + C_module(p) \Rightarrow E_project(T)`，其中 `M` 对应 `CODE_MAP.md`。
- PDF 示例中出现 `repo/CLAUDE.md`、模块级 `services/payment/CLAUDE.md`、`services/api/CLAUDE.md`，以及 `CODE_MAP.md` 中的 `auth/`、`payments/`、`billing/`、`orders/`、`notifications/`、`infra/` 等模块。

可核图示/结构：

- 页面图示显示 `CLAUDE.md 分层配置与自动加载`，表达 agent 进入不同目录时会叠加不同层级上下文。
- Code Map 页面用左右对比表达：没有 Code Map 时，Claude 需要自行搜索模块边界；有 Code Map 后，可以先看到业务模块概览。
- 项目结构图把顶层 `CLAUDE.md`、顶层 `CODE_MAP.md` 和模块级 `CLAUDE.md` 放在同一张结构图中。
- Harness 金字塔图把 `CLAUDE.md` 放在底层，上面依次是 Hooks、Skills、Plugins、MCP Servers + Subagents，说明基础上下文优先于高级自动化。

分析推断：

- PDF 的直接场景是大型代码库；该 idea 将 `CLAUDE.md` 扩展为“协作规则文件”，将 `CODE_MAP.md` 扩展为“工作地图文件”，可迁移到政策资料、数据指标体系、运维系统和论文阅读任务。
- 对普通工作者而言，最有 PPT 价值的部分是“先建地图和规则，再让 agent 执行”，技术组件名只作为例子出现。
- `S_map(T) << S_full(T)` 可解释为：一页地图无法替代完整材料，却能帮助 agent 快速找到该看的区域。
- PDF 文本抽取有字体映射警告，本文件依据文本搜索结合页面渲染核验公式和图示。

## 风险与边界

- 地图过旧会误导 agent。地图文件必须有维护责任人和更新时间。
- 地图过粗会漏掉关键边界；地图过细会占用上下文并增加维护成本。
- Agent 依据地图缩小范围后，仍可能漏查跨模块影响，尤其是权限、资金、合规、生产系统变更。
- 涉及税务、合规、隐私、资金、生产操作时，agent 只能辅助整理和检查，人类负责最终判断。
- MCP、Plugins、Hooks 等自动化能力会扩大 agent 的操作范围，必须先明确权限、审计和回滚机制。
- 代码库中的 LSP 能提供语义定位；普通文档任务需要依赖目录、标签、索引、知识库或人工整理的地图来替代。

## PPT 使用建议

适合放在“长程任务方法”页，作为一张跨角色方法论页。

建议页面主图：

- 左侧：一堆杂乱材料，标注“完整材料集 \(S_{full}(T)\)”。
- 中间：一张地图，标注“工作地图 \(S_{map}(T)\)”。
- 右侧：agent 按“全局规则 + 模块地图 + 当前任务”执行，并在检查点停下让人确认。

建议讲法：

- 程序员的例子：`CLAUDE.md` 是项目规则，`CODE_MAP.md` 是代码地图。
- 税务管理员的类比：规则文件是口径边界，地图文件是政策主题索引。
- 数据分析师的类比：规则文件是指标定义，地图文件是数据表和报表关系。
- 平台运营工程师的类比：规则文件是处置边界，地图文件是系统、告警和工单关系。
