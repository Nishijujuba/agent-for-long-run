# 子代理不是越多越好：关键是共享状态、任务所有权和交接协议

## 状态

confirmed

## 一句话主张

多 agent 可以扩大探索半径，也会放大协调成本；让它们真正适合长程任务，需要明确任务所有权、共享状态、证据回传和合并协议。

## 适用角色

程序员、算法工程师、数据分析师、平台运营工程师，也适合多资料综述、方案评审和复杂内容生产。

## 工作场景

一个主 agent 要读多份 PDF、理解仓库规则、生成 ideas，再让另一个 agent 基于 ideas 生成 PPT。它可以派子代理分别负责资料盘点、代码仓规则抽取、失败模式总结、PPT 价值评估。但如果没有共享白板和交接格式，多个 agent 会重复工作、互相覆盖、产生冲突结论。

## 讲给 PPT agent 的内容说明

这一页要避免把 subagent 讲成“开更多分身就会更快”。真正的价值是上下文隔离和局部探索：父代理保留主线，子代理去处理高噪声任务，例如读大量资料、搜索代码库、找反例、做安全审查、生成候选方案。父代理只接收结构化 evidence packet，再决定是否合并。

可以用“共享白板”类比：上面写着谁领取了什么任务、依据是什么、产物在哪里、哪些结论已验证、哪些存在冲突、下一步由谁接手。没有这块白板，多 agent 只是把一个人的混乱变成一群人的混乱。

## 应掌握的概念和方法论

- Orchestrator：保留总目标和决策权，负责派发、合并、停止。
- Isolated context：子代理只拿局部资料，避免主上下文被污染。
- Shared state：记录任务状态、负责人、证据、风险、阻塞和下一步。
- Claim / lease：子代理领取任务后有所有权和超时机制，避免重复执行。
- Evidence packet：子代理回传结论时附来源、关键证据、验证结果和不确定性。
- Merge protocol：多个产物合并前检查重复、冲突、覆盖范围和证据强度。

## 常见失败方式

第一种失败是并行失控。多个 agent 同时改同一批文件，没有锁、分支、任务领取记录，最后产生不可审查 diff。

第二种失败是摘要幻觉。子代理回传的摘要可能遗漏边界、误读来源或把推断写成事实；父代理不能把摘要当权威证据。

第三种失败是中央瓶颈。所有子代理产物都堆回父代理，但没有结构化字段，父代理又被长摘要压垮。
