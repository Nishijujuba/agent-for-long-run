# 子代理和 Swarm 的关键不是数量，而是共享状态与交接协议

## 状态

confirmed

## 一句话主张

多 agent 能扩大探索半径，也会放大协调成本；真正缺的不是更多 agent，而是清晰的任务所有权、共享状态、证据回传和合并协议。

## 适用角色

程序员、算法工程师、数据分析师、平台运营工程师；也适合多资料综述和多方案评审任务。

## 工作场景

一个主 agent 要完成长程任务：读多份 PDF、理解仓库规则、提炼 ideas、避免重复、保证来源可追溯。它可以派子代理分别负责资料搜索、代码仓阅读、失败模式归纳、PPT 风格评审。但如果没有共享状态和边界，多个子代理会重复工作、互相覆盖、产生冲突结论。

## 讲给 PPT agent 的内容说明

这一页要讲清楚“subagent 不是魔法分身”。它的核心价值是上下文隔离：父代理保留主线，子代理去做局部高噪声任务，比如大范围读仓库、读资料、找反例、做安全审查、生成候选方案。父代理只接收摘要、证据和建议，再决定下一步。

Swarm 的难点是协调。可以把缺失的 primitive 讲成一块共享白板：谁正在做什么、依据是什么、结论是否已验证、下一步由谁接手、哪些内容不能碰。没有这块白板，多 agent 只是把一个人的混乱变成一群人的混乱。

## 实际例子

课程笔记中，subagent 像被派去档案室的研究员。父代理不需要背下所有文件细节，只需要知道与当前决策相关的事实：项目里已有 lesson progress 记录、完成时间戳、dashboard 入口、可能的积分落点。这样既保留探索广度，又避免主会话过早进入上下文压力区。

对应到本仓库，一个子代理可以只负责 `resources/` 资源盘点，一个子代理只负责 `README.md` 和 `AGENTS.md` 规则抽取，一个子代理只负责失败模式清单。父代理再把这些结果整理成 `ideas/` 文件。高风险结论需要回到 PDF、代码或测试证据中抽查。

## 应掌握的概念和方法论

- Orchestrator：保留任务主线，决定派发、合并、停止。
- Isolated context：子代理只拿局部资料，避免主上下文被污染。
- Shared state：任务白板，记录任务状态、负责人、证据、风险和下一步。
- Claim / lease：子代理领取任务后有所有权和超时机制，避免重复执行。
- Evidence packet：子代理回传结论时附上来源、关键片段、验证结果和不确定性。
- Merge protocol：多个 agent 产物合并前要检查冲突、重复和覆盖范围。

## 常见失败方式

第一种失败是 swarm 并行失控。多个 agent 同时改同一批文件，没有锁、没有分支、没有任务领取记录，最后产生不可审查 diff。

第二种失败是摘要幻觉。子代理回传的摘要可能遗漏边界、误读来源或把推断写成事实。父代理不能把摘要当作权威证据。

第三种失败是中央瓶颈。所有子代理结果都堆回父代理，但没有结构化字段，父代理又被长摘要压垮。

## PPT 使用建议

适合做“分工与并行”页。画面可以是父代理在中间，四个子代理围绕：资料探索、代码导航、反例评审、并行执行。底部画一块共享白板，写任务状态、证据、阻塞、下一步。不要把这一页讲成“多开几个 agent 就会更快”，重点是协调机制。
