# 目标状态闭环：长程任务靠深度推进

## 状态

candidate

## 一句话主张

长程任务的难点是保持目标、状态和验证链不断线；agent 每一步都应围绕目标选择行动、记录观察、更新状态，并在关键检查点让人确认。

## 适用角色

跨角色，优先适用于程序员、数据分析师、平台运营工程师；也可迁移到税务管理员的政策口径整理、材料清单生成和风险提示任务。

## 工作场景

平台运营工程师或程序员要排查一个持续数小时的性能问题：agent 分批读取监控、日志、工单、代码和历史记录，维护一份“任务状态台账”，提出下一步排查动作，记录每次观察结果，并在涉及生产系统、结论定性或改动建议时交给人确认。

## Agent 协作方式

人负责给出目标 \(G\)、业务边界、资料范围和验收标准。Agent 负责维护当前状态 \(S_t\)，包括已读材料、已确认判断、未决问题、证据链接、风险点和下一步计划。每一步由 agent 选择行动 \(a_t\)，例如检索、阅读、对比、运行检查、生成草稿；行动后把观察结果 \(o_t\) 写回状态台账，形成下一轮状态 \(S_{t+1}\)。

可用一个通俗公式表达：

\[
a_t=\pi(G,S_t), \quad S_{t+1}=U(S_t,a_t,o_t)
\]

含义是：下一步行动来自“目标 + 当前状态”，行动结果必须更新任务状态。类比办一个跨部门事项：每次沟通之后都要更新事项清单、证据、责任人和下一步，否则时间越长越容易跑偏。

## 长程任务设计要点

- 目标：先写清最终交付物，例如“定位性能瓶颈并给出可验证改进建议”。
- 资料：把日志、监控、代码、工单、政策文件或数据字典列成输入清单。
- 计划：把任务拆成可检查阶段，例如收集证据、提出假设、验证假设、形成结论、生成报告。
- 检查点：每个阶段结束时让人确认“目标是否变化、证据是否足够、下一步是否值得做”。
- 记忆：用状态台账保存关键结论、排除项、待验证假设和人类确认记录，避免只依赖聊天上下文。
- 工具：agent 可以调用搜索、代码阅读、日志分析、数据分析、文档生成等工具；涉及生产操作或合规判断时必须先获得人类授权。
- 验收标准：每个结论要能追到证据；每个建议要能说明影响范围、验证方法和回滚方式。

## 风险与边界

- 强模型也可能在长任务中跑偏、遗忘、误判证据或过早收敛；状态台账和检查点是必要护栏。
- Agent 写入状态台账的内容也可能出错，人类需要抽查关键证据，尤其是税务、合规、隐私、生产系统和客户影响相关事项。
- PDF 的主要例子偏 AI Coding 和性能优化；迁移到税务、运营、数据分析等场景属于分析推断，需要用更贴近听众的工作案例重新包装。
- 单靠更长上下文无法保证长程任务质量；任务还需要资料组织、工具权限、阶段验收和责任边界。
- 涉及公式时，PPT 只保留直观解释：目标像任务终点，状态像工作台账，行动像下一步办理，观察像办理结果。

## PPT 使用建议

适合放在“长程任务方法”页，作为全场方法论核心页。页面主图可画成：

目标 \(G\) -> 当前状态 \(S_t\) -> 下一步行动 \(a_t\) -> 观察结果 \(o_t\) -> 更新后的状态 \(S_{t+1}\)

旁边配一个普通工作例子：平台运营工程师排查线上性能问题，agent 每轮都更新“已确认事实、待验证假设、下一步动作、需要人确认的风险”。选用理由：相比模型排名、OPD、PMF 等偏技术内容，这个 idea 更容易转成普通工作者能执行的协作流程；代价是弱化 GLM-5.1 的模型宣传信息，把 PDF 中的技术讨论转成通用 agent 工作法。

口播可把公式压成一句话：目标 \(G\) 是终点，状态 \(S_t\) 是台账，行动 \(a_t\) 是下一步办理，观察 \(o_t\) 是办理结果。
