# 状态外化：长程 Agent 要把关键记忆写出上下文

## 状态

candidate

## 一句话主张

Agent 的“失忆”常来自长历史的成本与保真约束；长程任务要把目标、事实、决策、待办和验收标准外化成文件与检查点，让 agent 每一轮都能回到可靠状态。

## 适用角色

跨角色：税务管理员、程序员、数据分析师、算法工程师、平台运营工程师。

## 工作场景

一个知识工作者让 agent 连续处理多小时或多天的资料密集型任务，例如：

- 税务管理员整理多份政策口径、生成风险提示和材料清单。
- 程序员根据需求修改代码、跑测试、同步变更说明。
- 数据分析师读取表格、口径文档和会议记录，产出分析报告。
- 平台运营工程师整理告警、日志、工单和复盘结论。

这些任务都会持续产生来源文件、工具调用结果、历史讨论、临时判断和待确认问题。若这些状态只留在聊天上下文里，agent 后续容易遗漏前提、混淆版本或重复劳动。

## Agent 协作方式

Agent 负责读取资料、提炼事实、维护外部状态文件、生成阶段产物和自检清单。人类负责设定目标、确认关键判断、授权敏感操作和验收最终结果。

推荐协作方式：

1. 任务开始时建立“任务卡”：目标、范围、角色、输出格式、禁止事项。
2. 资料读取后建立“来源地图”：每个事实来自哪个文件、哪一页或哪段记录。
3. 每次重要讨论后更新“决策日志”：已确认结论、放弃方案、原因。
4. 每个阶段结束前生成“检查点”：当前进度、剩余问题、下一步计划。
5. 最终交付前运行“验收清单”：事实来源、角色场景、风险边界、格式要求。

## 长程任务设计要点

- 目标：把长程任务拆成可检查阶段，避免把所有资料和所有要求一次性塞进上下文。
- 资料：原文、摘要、推断分开记录。无法核实的内容标成“待确认”。
- 计划：每一阶段只推进一个明确目标，例如“提炼候选 idea”“确认角色场景”“生成 PPT 页结构”。
- 检查点：每完成一个阶段就写下当前状态，下一轮 agent 先读检查点再继续。
- 记忆：关键状态进入外部文件或结构化清单，聊天记录只作为过程材料。
- 工具：使用 PDF 渲染、文本抽取、搜索、表格处理、测试命令等工具时，记录输入、输出和结论。
- 验收标准：每个交付物都要能回答“来源在哪里、谁会用、怎样落地、哪里有风险”。

可以用一个直观公式解释给普通听众：

\[
C_{\text{task}} \approx C_{\text{model}}(L) + C_{\text{memory}}(M) + C_{\text{tool}}(R)
\]

其中 \(L\) 是上下文长度，\(M\) 是记忆管理成本，\(R\) 是工具调用记录。PDF 中还展示了 full attention 的 \(O(L^2)\) 成本直觉：当上下文长度变长，模型在历史信息里找相关内容会变得更贵。对使用者来说，实用结论是：要管理工作状态，减少无效上下文，让 agent 把注意力放在当前阶段真正需要的信息上。

## 风险与边界

- 摘要压缩会丢细节，尤其容易丢掉例外条件、版本差异和少数派证据。
- 外部记忆文件也可能过期，必须标注更新时间、来源和责任人。
- 涉及税务、合规、隐私、生产系统操作时，agent 只能辅助整理、提示和检查，人类需要承担最终判断与授权责任。
- 长上下文能力提升无法替代验收标准。上下文放得更多，错误也可能传播得更远。
- 若来源文件本身有误，agent 的整理结果也会继承错误，因此关键事实需要回到原文核查。
