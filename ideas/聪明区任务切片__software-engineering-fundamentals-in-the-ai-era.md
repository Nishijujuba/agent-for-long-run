# 把长程任务切进 AI 的聪明区

## 状态

candidate

## 一句话主张

长程任务交给 agent 前，人要先写清“目的地”、拆成可独立验收的垂直切片，并在新上下文中复核，让 agent 每次处理上下文可控、反馈明确的小任务。

## 适用角色

跨角色。程序员最直接，也可迁移到税务管理员、数据分析师、算法工程师、平台运营工程师等需要把复杂工作拆成阶段产物的人。

## 工作场景

一个程序员要让 agent 完成一项中等规模功能改造，例如新增积分系统；对应到普通知识工作，可以类比为税务管理员整理一套政策适用口径、数据分析师完成一份专题分析报告、平台运营工程师把一次告警复盘变成自动巡检流程。

共同点是：任务都包含资料收集、方案判断、拆分执行、质量复核和最终交付，无法靠一次提问稳定完成。

## Agent 协作方式

人负责战略层：定义问题、确定边界、批准取舍、判断风险、验收最终质量。PDF 中可读图示明确表达了 “Strategic level = You, Tactical level = AI”。

Agent 负责战术层：根据目的地文档提出问题、补齐信息、拆分垂直切片、执行单个切片、运行检查、提交复核材料。

协作时不要把整包任务一次性塞给 agent。更好的做法是把任务切成“薄工单”：每个工单有输入、输出、依赖、验收标准和失败处理。直观类比是让同事一次处理一张清楚的工单，避免把一整箱资料倒在桌上。

## 长程任务设计要点

1. 先写目的地文档

   目的地文档回答“最后要交付什么”。在软件任务里可以是 PRD；在税务、分析、运营任务里可以是任务说明书。它至少包括问题陈述、解决方案、用户故事或使用场景、关键决策、测试或核验办法、拟修改模块、范围外事项和完成定义。

2. 把任务切成垂直切片

   PDF 的 `PRD-to-issues` 示例把 PRD 拆成可独立抓取的 issues，并强调 vertical slices / tracer bullets。通俗说，切片要尽量从输入走到可见输出，例如“查政策依据 -> 形成口径 -> 生成材料清单 -> 人工复核”，避免只做孤立的一层资料整理。

3. 明确输入资料与工具

   输入资料包括目的地文档、来源文件、工单、阶段摘要和待确认问题；工具包括检索、测试、类型检查、引用回查、人工审批表或 QA 清单。这样每个切片都有材料来源和核验手段。

4. 控制上下文，避免掉进愚蠢区

   PDF 用图展示 `Smart zone` 和 `Dumb zone`，在约 100k tokens 附近质量开始明显下降，并给出关系数量公式：

   \[
   R(n)=\binom{n}{2}=\frac{n(n-1)}{2}\approx \frac{n^2}{2}
   \]

   直觉是：资料越多，模型需要同时处理的关系按平方级增加，遗漏和混淆会变多。长任务要靠阶段摘要、工单文件、检查点和清理上下文来维持质量。

5. 每个阶段都要有反馈回路

   对程序员来说是测试、类型检查、代码审查和 QA；对非程序员来说是样本核验、口径清单、数据重算、引用回查和人工审批。没有反馈回路，agent 会越做越像“看起来完成”。

6. 复核最好放到新上下文

   PDF 的 review/QA 页展示了同一上下文 review 会积累高 token，`/clear` 后复核能回到低 token 区间。分析推断是：实现者和复核者应尽量分开上下文，减少自我确认和历史包袱。

  - 第 6 部分标题为“目的地文档：PRD”。可读代码块 `write-a-PRD` 包含问题描述、仓库探索、访谈补洞、PRD 模板、测试决策、拟修改模块、范围外事项和完成定义。

  - 第 8 部分标题为“实现：Ralph 循环、反馈回路与 TDD”。可读 `once.sh` 示例把 issues、recent commits 和 prompt 一次性交给 agent，让它挑选下一个任务并完成。

- 分析推断：
  - PDF 以软件工程为主，PPT 可把 PRD 泛化为“目的地文档”，把 TDD/QA 泛化为“可执行检查与人工复核”，把 vertical slice 泛化为“从输入到可验收输出的小闭环”。
  - 该 idea 适合放进面向普通工作者的长程任务方法页，因为它解释了 agent 协作失败的常见根因：目标不清、上下文过大、切片过粗、缺少验收。

## 风险与边界

- 目的地文档写得含糊，agent 会把错误目标执行得更快；这在代码、税务、数据分析和运营流程里都会放大风险。
- 垂直切片如果缺少验收标准，agent 可能只交付表面结果，例如代码能运行、报告有图、清单看起来完整，却没有满足业务判断。
- 上下文越大，模型越容易漏掉关键约束；长上下文不能替代阶段性总结、文件化记录和复核关卡。
- 生产系统、税务合规、隐私数据、客户承诺等高风险任务必须由人批准关键决策。Agent 可以辅助检索、整理、执行和自检，责任仍在人。
- 本 PDF 在当前环境下中文正文存在字体/映射缺失，证据主要来自可恢复目录、可读英文图表、公式、代码块和页面结构；精确中文原句需要在能完整渲染中文的环境中复核。

## PPT 使用建议

适合放在第 9 页“长程任务方法”，也可作为程序员案例页的核心方法。

建议画面：左侧放简化版 `Smart zone / Dumb zone` 曲线，说明任务越厚越容易让 agent 质量下降；右侧放流程：

目的地文档 -> 垂直切片 -> 单切片执行 -> 自动检查 -> 新上下文复核 -> 人类验收

演讲口径：agent 像战术执行同事，擅长处理清楚、短程、可检查的工单；人要做的是把长程任务设计成一串可验收的小闭环。
